# BugWorld 项目重构指导文档

## 项目背景
这是一个名为"虫群世界(BugWorld)"的网页游戏项目，当前使用jQuery技术栈，需要重构为Vue3 + TypeScript架构。

## 核心原则（必须严格遵守）

### 1. 功能保真原则 - 最高优先级
- **绝对不能修改或删除任何已实现的游戏功能和逻辑**
- **必须100%还原现有jQuery版本的所有功能**
- 重构的唯一目的是技术栈迁移，不是功能改进
- 任何功能实现必须与原代码行为完全一致
- 如果发现原代码的bug或问题，必须先询问我是否需要修复

### 2. 协作模式 - 辅助而非主导
- **你的角色是技术顾问和辅助开发者，不是项目主导者**
- **任何代码改动前必须先向我说明并征求同意**
- 不要擅自推进开发进度
- 提供建议时给出多个方案供我选择
- 等待我的明确指示后再执行具体操作

### 3. 代码风格原则
- **保留中文命名变量作为项目特色**（如：`虫群`、`属性`、`设施`等）
- 中文命名是项目的特色标识，不要建议改为英文
- 注释可以使用中文
- 类型定义和接口可以使用中文属性名

## 技术要求

### 目标技术栈
- Vue 3 (Composition API)
- TypeScript
- Vite（替代Webpack）
- Pinia（状态管理）
- 保留Express作为开发服务器

### 架构设计要求

#### 1. 高度组件化
- 每个游戏UI元素都应该是独立的Vue组件
- 组件必须支持插槽(slots)和动态配置
- 为未来的创意工坊功能预留扩展接口
- 组件间通信使用明确的props/emits，避免隐式依赖

#### 2. 模块化设计
保持原项目的模块结构：
- `Object/` - 游戏核心对象类（BugNest、Bug、Facility等）
- `State/` - 状态管理相关
- `Tiles/` - UI组件（对应Vue组件）
- `Modules/` - 功能模块
- `library/` - 数据和工具函数

#### 3. 创意工坊支持
- 设计可扩展的插件系统架构
- 游戏数据（虫群类型、设施、事件等）应该支持JSON配置
- 提供清晰的扩展接口文档结构
- 考虑未来支持用户自定义内容

## 现有功能清单（必须全部实现）

### 核心对象系统
- Area（区域）
- BugNest（虫巢）- 包含属性、参数、系数、虫群、设施、状态等
- Bug（虫群单位）- 虫后、工虫、兵虫等
- Facility（设施）- 虫母室、孵化室等
- Work（工作）- 觅食、探索、哺育、修建设施
- Event（事件）- 随机事件系统
- Characteristic（特性）
- State（状态系统）- Effect、Entry、Hidden、Impact、Movement、Status

### UI组件系统（Tiles）
- logTile - 日志栏
- bugNestTile - 虫巢信息栏
- bugGroupTile - 虫群信息栏
- workTile - 工作信息栏
- orderTile - 命令信息栏
- facilityTile - 设施信息栏
- eventTile - 事件信息栏
- buildTile - 建设信息栏
- chooseTile - 选择界面

### 游戏机制
- 虫群加入虫巢系统
- 工作解锁和分配系统
- 设施建造和管理
- 事件触发和处理
- 状态和特性系统
- 资源生产消耗计算

## 工作流程要求

### 第一阶段：分析和规划
1. 仔细阅读现有代码，理解每个模块的功能
2. 向我提交详细的重构方案，包括：
   - 目录结构设计
   - 组件拆分方案
   - 状态管理方案
   - 类型定义结构
3. **等待我审核批准后再进入下一阶段**

### 第二阶段：基础架构搭建
1. 搭建Vue3 + Vite项目基础
2. 配置TypeScript
3. 设置Pinia状态管理
4. 建立基础目录结构
5. **每个步骤完成后向我汇报，等待确认**

### 第三阶段：逐模块迁移
1. 按照我指定的优先级顺序迁移模块
2. 每迁移一个模块：
   - 先向我说明迁移计划
   - 实现后进行功能对比测试
   - 向我展示迁移结果
   - 等待我确认后再继续下一个
3. 优先迁移核心对象系统，再迁移UI组件

### 第四阶段：测试和验证
1. 确保所有原有功能正常工作
2. 对比新旧版本的行为一致性
3. 提供测试报告供我审查
